# Quickstart: Interactive Guitar Fretboard Visualizer

This guide validates the feature end-to-end once implemented. It intentionally omits
implementation code — see `data-model.md` and `contracts/` for the shapes each check below
depends on, and `tasks.md` (generated by `/speckit-tasks`) for build-out steps.

## Prerequisites

- A reasonably current Node.js (for running the theory unit tests; Node 18+ for built-in
  `node --test`).
- Any static file server for local browser testing — opening `index.html` directly via `file://`
  can block native ES module `import`s and cross-origin fetches in some browsers, so serve the
  repo root instead, e.g.:
  ```sh
  npx serve .
  # or
  python3 -m http.server 8000
  ```
  This is a dev-convenience step only — it is not a build step and nothing is compiled or bundled.
- A modern evergreen browser (Chrome, Firefox, Safari, or Edge) with network access on first load
  (to fetch the `soundfont-player` CDN script and the FluidR3_GM guitar samples).

## 1. Run the music-theory unit tests (hard gate)

```sh
node --test tests/theory.test.js
```

**Expected outcome**: all tests pass, covering — per the constitution's hard-gate requirement —
open strings, 12-fret octave wraparound, enharmonic equivalents, every tuning in `TUNINGS`, and
edge frets (0, 12, 24). A failing or missing test here blocks the feature from being considered
complete (constitution Principle IV).

## 2. Load the app and validate the base fretboard (Story 1)

1. Serve the repo root and open `index.html` in a browser.
2. **Confirm**: strings are ordered high-E (string 1) at top, low-E (string 6) at bottom.
3. **Confirm**: fret markers appear at 3, 5, 7, 9, 12, 15, 17, 19, 21, 24, with double dots at 12
   and 24.
4. **Confirm**: open-string notes render distinctly to the left of fret 1, past the nut line.

## 3. Validate tuning selection (Story 2)

1. Open the tuning selector; confirm the "D-Family," "G-Family," and "C-Family" groups each list
   their exact named tunings from the spec, plus "Custom Tuning."
2. Select **Drop D**. **Confirm**: string 6 (low-E) now reads D at fret 0, and every fretted note
   on that string recalculates accordingly.
3. Select **Custom Tuning**, set each string independently, and confirm the fretboard updates to
   match.

## 4. Validate root, scale, and diatonic coloring (Stories 3–5)

1. Select root **C**, scale **Ionian (Major)**.
2. **Confirm**: C, D, E, F, G, A, B render in their assigned dark color-role; C#/Db, D#/Eb, F#/Gb,
   G#/Ab, A#/Bb render with no color (Story 5, Scenario 1).
3. **Confirm**: focal point defaults to C, and C/E/G render bright + bordered as a major triad
   (Story 5, Scenario 2).
4. Click **E**. **Confirm**: focal point becomes E, and E/G/B render bright + bordered, correctly
   shown as a **minor** triad (Story 5, Scenario 3) — quality is computed, not assumed.
5. With focal point still E, toggle off G and toggle on A. **Confirm**: the bright set becomes
   E/A/B (Story 5, Scenario 5), and attempting to toggle on F# is unavailable (Scenario 6) because
   F# is not diatonic to C Major.
6. Switch scale to **Lydian** (same root C). **Confirm**: F# becomes an available toggle for focal
   point E (Story 5, Scenario 7), and the whole diatonic color set updates with no stale notes left
   from Ionian (Story 4, Scenario 2).

## 5. Validate label modes (Story 6)

1. With any key/scale active, confirm every fret position shows a note-name base layer at all
   times, in a neutral non-distracting style — this must never disappear.
2. Switch label mode to **Degrees**. **Confirm**: diatonic notes show scale-degree numbers
   relative to the root (e.g. focal point E in C Major shows "3", "5", "7").
3. Switch to **Intervals**. **Confirm**: the same notes show interval shorthand (e.g. "R", "M3",
   "P5", "m7").
4. **Confirm**: in every label mode, the root is distinguished by both color-role AND a secondary
   border/shape indicator.

## 6. Validate the fret-range slider (Story 7)

1. On load (first-ever visit, no stored settings), confirm the left handle reads "N" and the right
   handle reads "24," with all frets visible.
2. Drag the left handle to fret 5. **Confirm**: label updates to "5," and frets 0–4 hide.
3. Drag the right handle to fret 12. **Confirm**: label updates to "12," and frets 13–24 hide.
4. Attempt to drag either handle past the other. **Confirm**: the drag is constrained; the range
   never inverts or collapses to zero frets.

## 7. Validate audio playback (Story 8)

1. Click/tap several fret positions across different strings and octaves.
2. **Confirm**: each produces an audible, correctly-pitched real guitar tone (not a synthesized
   beep) — cross-check open low-E vs. 12th-fret high-E sound at distinct octaves, not identical.
3. Trigger several notes in rapid succession (e.g. a fast scale run). **Confirm**: each plays
   cleanly with no improper cutoff or lag.
4. Disconnect from the network after the first successful load, then reload and replay previously
   triggered notes. **Confirm**: playback still works (samples were cached, not re-fetched).
5. Temporarily block the audio CDN (or simulate a slow/broken connection) before first load.
   **Confirm**: a visible, non-blocking error indicator appears, and the rest of the UI remains
   fully interactive.

## 8. Validate the capo (Story 9)

1. With standard tuning and no capo, place a capo at fret 3. **Confirm**: frets 0–2 become
   unplayable, and the fret-range slider's left handle locks to fret 3, labeled "Capo."
2. With **Absolute** label mode active, confirm the capo position on the A string (5th string)
   displays as "C."
3. Switch to **Relative** label mode. **Confirm**: that same physical position now displays as
   "A," and fretting 2 more frets above the capo (physical fret 5) displays as "B" in Relative mode
   while Absolute mode simultaneously shows "D" for the same physical position.
4. **Confirm**: audio for any fret always plays the true Absolute-mode pitch, regardless of which
   label mode is currently displayed.
5. Set capo back to 0. **Confirm**: Absolute and Relative modes now produce identical labels
   everywhere, and the fret-range slider's left handle is freely draggable again, defaulting to
   "N."

## 9. Validate persistence (Clarifications session, FR-039/FR-040)

1. Change tuning, root, scale, focal point/chord-tone overrides, label mode, fret range, and capo
   away from defaults.
2. Reload the page (plain refresh, not an explicit reset control). **Confirm**: every one of the
   above settings is restored exactly as left — no reset to defaults.
3. Inspect the `fret-navigator-settings` key in the browser's `localStorage` dev tools panel.
   **Confirm**: it contains a `schemaVersion` field and the fields listed in
   `contracts/settings-schema.md`.

## Done

If all sections above pass, the feature satisfies its spec's acceptance scenarios and success
criteria end-to-end.
